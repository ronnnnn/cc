# Codex MCP 委譲パターン

コーディングタスクを Codex MCP に委譲する際の詳細手順。プラグイン定義 agent では MCP ツールが利用不可のため、general-purpose subagent を経由して MCP ツールにアクセスする。

## Codex MCP ツール

| ツール                    | 用途                               |
| ------------------------- | ---------------------------------- |
| `mcp__codex__codex`       | 新しい Codex セッションを開始      |
| `mcp__codex__codex-reply` | スレッド ID で既存セッションを継続 |

**重要:** subagent は `mcp__codex__codex` を呼び出す前に、ToolSearch ツールで `codex` を検索してツールをロードする必要がある。

## subagent プロンプトの構築

SKILL.md のステップ 4 でコーディングタスクの subagent を起動する際、以下の構造のプロンプトを構築する。`<...>` はプレースホルダーで実際の値に置換する。

### プロンプト構造

subagent に渡すプロンプトは 3 つのセクションで構成する:

1. **設計書** - ステップ 2 で作成したファイル単位の変更設計
2. **実行手順** - 以下の 5 ステップ
3. **制約事項** - Codex の自律性と報告要件

### 制約事項 (subagent プロンプトに含める内容)

- 設計書はガイダンスであり、より良いアプローチがあれば変更してよい
- 設計変更を行った場合は理由と内容を必ず報告すること
- プロジェクトルール (`developer-instructions` で渡した内容) を厳守すること
- 既存のテストが壊れないことを確認すること
- テスト作成が不要なタスク (設定ファイル変更、ドキュメント生成等) ではテスト作成ステップをスキップする

### 実行手順 (subagent が実行する内容)

#### 手順 1: プロジェクトルールの収集

Glob ツールで以下のパスを検索し、存在するルールファイルを全て Read ツールで読み込む:

- `<プロジェクトルート>/.claude/rules/` 配下
- `~/.claude/rules/` 配下

ルールファイルが存在しない場合はスキップする。読み込んだルールは、ユーザールール、プロジェクトルールの順で改行で結合して 1 つのテキストにまとめる。これにより、テキスト末尾に記述されるプロジェクト固有のルールが優先されることが期待される。ルール同士が競合する場合は、常にプロジェクトルールを優先して解釈すること。

#### 手順 2: Codex にコーディングを委譲

ToolSearch ツールで `codex` を検索して `mcp__codex__codex` ツールをロードした後、以下のパラメータで呼び出す:

| パラメータ               | 値                        | 説明                                           |
| ------------------------ | ------------------------- | ---------------------------------------------- |
| `prompt`                 | 設計書 + コーディング指示 | 設計書の内容を含め、設計変更を許容する旨を明記 |
| `developer-instructions` | プロジェクトルール        | 手順 1 で収集したルール (なければ省略)         |
| `sandbox`                | `workspace-write`         | ワークスペース内の書き込みを許可               |
| `cwd`                    | `<プロジェクトルート>`    | 作業ディレクトリ                               |
| `approval-policy`        | `on-failure`              | 失敗時のみ承認を要求                           |

`prompt` に含める指示:

- 設計書はファイル単位の変更ガイダンスであり、より良いアプローチがあれば設計変更してよい
- 設計変更を行った場合は何をなぜ変更したか報告すること
- 各ファイルの変更内容を具体的に記述

#### 手順 3: Codex の結果を確認

Codex の出力を確認する:

- 設計書の要件を満たしているか
- エラーや警告がないか
- 設計変更がある場合、その妥当性

問題がある場合は `mcp__codex__codex-reply` で追加指示を送る (`threadId` に手順 2 のスレッド ID を使用)。

#### 手順 4: テスト作成を委譲

`mcp__codex__codex-reply` で以下を指示する:

- 実装したコードに対するテストを作成する
- 既存のテストパターンやフレームワークに従う
- テストを実行して全テストがパスすることを確認する

`threadId` には手順 2 で取得したスレッド ID を使用する。

#### 手順 5: 結果の報告

以下の情報を含めてマークダウン形式で報告する:

- タスク名とステータス (完了 / 部分完了 / 失敗)
- Codex スレッド ID
- 設計変更の有無と内容
- 変更ファイル一覧
- テスト結果 (テストファイルパス、パス/失敗件数)
- 備考

## エラーハンドリング

### Codex セッション開始に失敗した場合

subagent 自身がコーディングを実行するフォールバックに切り替える。結果報告に「Codex 利用不可のため subagent が直接実装」と明記する。

### Codex がタスクを完了できなかった場合

`mcp__codex__codex-reply` で最大 2 回追加指示を送る。それでも完了できない場合:

1. Codex の出力と失敗理由を記録する
2. subagent 自身でのフォールバック実装を試みる
3. 結果報告に経緯を明記する

### テスト作成に失敗した場合

コーディング結果は保持し、テスト作成失敗のみを報告する。メインセッションで対応を判断する。
