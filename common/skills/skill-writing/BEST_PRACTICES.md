## 1. コア原則

### 1.1 簡潔さが重要

**原則**: コンテキストウィンドウは公共の財産

スキル内のすべてのトークンにコストがあるわけではありません：

- **起動時**: すべてのスキルの `name` と `description` のみが事前ロード
- **トリガー時**: Claude は関連するスキルの SKILL.md のみを読む
- **必要時**: 追加ファイルは必要に応じてのみロード

しかし、SKILL.md で簡潔であることは依然として重要です。Claude がそれをロードすると、すべてのトークンが会話履歴および他のコンテキストと競合します。

**デフォルトの仮定**: Claude はすでに非常に賢い

Claude が既に持っていないコンテキストのみを追加してください。情報の各部分に異議を唱えてください：

- [ ] 「Claude は本当にこの説明が必要ですか？」
- [ ] 「Claude がこれを知っていると仮定できますか？」
- [ ] 「このパラグラフはそのトークンコストを正当化していますか？」

**良い例：簡潔** （約 50 トークン）：

````markdown
## PDF テキストの抽出

テキスト抽出には pdfplumber を使用します：

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
````

**悪い例：冗長すぎる** （約 150 トークン）：

```markdown
## PDF テキストの抽出

PDF（Portable Document Format）ファイルは、テキスト、画像、およびその他のコンテンツを含む一般的なファイル形式です。PDF からテキストを抽出するには、ライブラリを使用する必要があります。PDF 処理に利用できるライブラリは多くありますが、使いやすく、ほとんどのケースに対応しているため、pdfplumber をお勧めします。まず、pip を使用してインストールする必要があります...
```

簡潔なバージョンは、Claude が何が PDF であるか、およびライブラリがどのように機能するかを知っていることを前提としています。

### 1.2 適切な自由度を設定する

特異性のレベルをタスクの脆弱性と変動性に合わせてください。

**高い自由度**（テキストベースの指示）：

**使用する場合**：

- 複数のアプローチが有効である
- 決定はコンテキストに依存する
- ヒューリスティックがアプローチをガイドする

**例**：

```markdown
## コードレビュープロセス

1. コード構造と組織を分析する
2. 潜在的なバグやエッジケースを確認する
3. 可読性と保守性の向上を提案する
4. プロジェクト規約への準拠を確認する
```

**中程度の自由度**（疑似コードまたはパラメータ付きスクリプト）：

**使用する場合**：

- 推奨パターンが存在する
- ある程度の変動が許容される
- 構成が動作に影響する

**例**：

````markdown
## レポート生成

このテンプレートを使用して、必要に応じてカスタマイズしてください：

```python
def generate_report(data, format="markdown", include_charts=True):
    # データを処理する
    # 指定された形式で出力を生成する
    # オプションで視覚化を含める
```
````

**低い自由度**（特定のスクリプト、パラメータなし、またはほぼなし）：

**使用する場合**：

- 操作は脆弱でエラーが発生しやすい
- 一貫性が重要である
- 特定のシーケンスに従う必要がある

**例**：

````markdown
## データベースマイグレーション

正確にこのスクリプトを実行してください：

```bash
python scripts/migrate.py --verify --backup
```

コマンドを変更したり、追加のフラグを追加したりしないでください。
````

**類推**: Claude をパスを探索するロボットと考えてください：

- **両側に崖がある狭い橋**: 前に進む安全な方法は 1 つだけです。具体的なガードレールと正確な指示を提供してください（低い自由度）。
- **危険のない開けた野原**: 多くのパスが成功につながります。一般的な方向を示し、Claude が最適なルートを見つけることを信頼してください（高い自由度）。

### 1.3 使用予定のすべてのモデルでテストする

スキルはモデルへの追加として機能するため、有効性は基盤となるモデルに依存します。

**モデル別のテスト考慮事項**：

- **Claude Haiku**（高速、経済的）: スキルは十分なガイダンスを提供していますか？
- **Claude Sonnet**（バランス型）: スキルは明確で効率的ですか？
- **Claude Opus**（強力な推論）: スキルは過度な説明を避けていますか？

Opus で完璧に機能するものは、Haiku ではより詳細が必要な場合があります。複数のモデルでスキルを使用する予定がある場合は、すべてのモデルでうまく機能する指示を目指してください。

## 2. メタデータ設計

### 2.1 name (必須)

**仕様**:

- [ ] 形式: `lowercase-hyphen-case`
- [ ] 文字種: 小文字の Unicode 英数字とハイフン (`-`) のみ
- [ ] 長さ: 64 文字以内
- [ ] **ディレクトリ名と完全一致**

**命名パターンの推奨**:

スキル名には **動名詞形**（動詞 + -ing）を使用することをお勧めします。これはスキルが提供するアクティビティまたは機能を明確に説明しているためです。

**良い命名例（動名詞形）**:

- `processing-pdfs`（PDF の処理）
- `analyzing-spreadsheets`（スプレッドシートの分析）
- `managing-databases`（データベースの管理）
- `testing-code`（コードのテスト）
- `creating-documentation`（ドキュメントの作成）

**許容される代替案**:

- 名詞句: `pdf-processing`, `spreadsheet-analysis`
- アクション指向: `process-pdf`, `analyze-spreadsheet`

**避けるべき**:

- 曖昧な名前: `helper`, `utility`, `tool`
- 過度に一般的: `document`, `data`, `file`
- スキルコレクション内の一貫性のないパターン

**チェックリスト**:

- [ ] スキル名が明確で説明的
- [ ] 既存のスキルと重複していない
- [ ] 省略形を避け、理解しやすい名前
- [ ] ディレクトリ名 = YAML frontmatter の `name`
- [ ] 一貫した命名パターンを使用

**良い例**:

```yaml
name: api-documentation-generator
name: react-component-builder
name: test-suite-creator
```

**悪い例**:

```yaml
name: APIDocGen  # 大文字が含まれている
name: api_doc    # アンダースコアは使用不可
name: tool       # 曖昧すぎる
```

### 2.2 description (必須)

**仕様**:

- [ ] 長さ: 200 文字以内推奨 (厳密な上限なし)
- [ ] 内容: **何をするか (What)** + **いつ使うか (When)**

**チェックリスト**:

- [ ] スキルの機能が明確
- [ ] 使用すべきタイミングが具体的
- [ ] Claude が判断できる情報を含む
- [ ] 専門用語は必要最小限

**テンプレート**:

```
[このスキルが何をするか]. [特定のトリガー条件やユーザー要求]のときに使用します。
```

**良い例**:

```yaml
description: JSDoc コメント付き TypeScript ソースコードから、リクエスト/レスポンス例と認証詳細を含む包括的な API ドキュメントを生成します。ユーザーが TypeScript または JavaScript プロジェクトの API ドキュメント作成または更新を依頼したときに使用します。
```

**悪い例**:

```yaml
description: コーディングを手伝います  # 曖昧、いつ使うか不明
description: ドキュメントを作成します  # いつ使うか不明
description: このスキルができることをあらゆるシナリオや状況において細かい詳細まですべて説明する非常に長い説明文で読みにくく素早く理解することが困難です...  # 長すぎる
```

### 2.3 allowed-tools (オプション)

**用途**: ユーザーの確認なしに自動実行を許可するツール

**チェックリスト**:

- [ ] 必要最小限のツールのみ許可
- [ ] セキュリティリスクを評価済み
- [ ] 破壊的操作 (削除、上書き) は慎重に

**例**:

```yaml
allowed-tools:
  - Read
  - Write
  - Bash
```

**注意**: 過度に許可するとセキュリティリスクが高まります。

## 3. SKILL.md 構造

### 3.1 必須要素

**チェックリスト**:

- [ ] YAML frontmatter が存在
- [ ] `name` フィールドが存在
- [ ] `description` フィールドが存在
- [ ] Markdown 本文が存在

**最小構成**:

```markdown
---
name: my-skill
description: このスキルが何をするか、いつ使うかを記述します。
---

# My Skill

## 概要

簡潔な説明

## 使用方法

このスキルの使い方

## 例

最低1つの具体例
```

### 3.2 推奨セクション構造

**基本構造**:

```markdown
# [スキル名]

## 概要

スキルの目的と主要機能

## 使用方法

### 前提条件

（必要な場合）

### 実行手順

1. ステップ 1
2. ステップ 2

## 例

### 例 1: [ユースケース]

具体的な例

### 例 2: [別のユースケース]

異なる例

## 追加リソース

（必要な場合）他のファイルへの参照

## トラブルシューティング

（推奨）よくある問題と解決策
```

### 2.3 セクションの順序

**推奨順序**:

1. **概要**: スキルの目的
2. **使用方法**: 基本的な手順
3. **例**: 具体的な使用例
4. **詳細設定**: 高度なオプション
5. **トラブルシューティング**: 問題解決
6. **参考情報**: 外部リンク、追加リソース

**理由**: 一般的な使用パターンから詳細へ、段階的に情報を提供

## 4. プログレッシブディスクロージャ

### 4.1 原則

**目的**: Claude が必要な情報だけを読み込むようにする

**チェックリスト**:

- [ ] SKILL.md は核心的な指示のみ
- [ ] 詳細情報は別ファイルに分離
- [ ] ファイル間の参照が明確

### 4.2 情報の階層化

**Level 1: SKILL.md (必須)**

- スキルの基本的な使い方
- 最も一般的なユースケースの例
- 他のリソースへのポインタ

**Level 2: REFERENCE.md (詳細)**

- API リファレンス
- 全パラメータの詳細
- 技術仕様

**Level 3: examples/ (具体例)**

- 複数の具体的な例
- 各例は個別のファイル

**Level 4: templates/ (テンプレート)**

- 再利用可能なテンプレート
- ボイラープレート

**Level 5: scripts/ (自動化)**

- ヘルパースクリプト
- 検証ツール

### 4.3 参照の書き方

**チェックリスト**:

- [ ] 相対パスを使用
- [ ] 参照先が存在することを確認
- [ ] 参照の目的を明記

**良い例**:

```markdown
詳細な API 仕様については `REFERENCE.md` を参照してください。

複数の例は `examples/` ディレクトリにあります:

- `examples/basic-usage.md`: 基本的な使い方
- `examples/advanced-patterns.md`: 高度なパターン
```

**悪い例**:

```markdown
詳細は REFERENCE.md を見てください。 # どんな詳細？
他のファイルも見てね。 # どのファイル？
```

### 4.4 深くネストされた参照を避ける

**原則**: **SKILL.md から 1 レベル深い参照を保つ**

参照されたファイルから参照されたファイルを部分的に読む場合があります。ネストされた参照に遭遇すると、Claude は `head -100` などのコマンドを使用してコンテンツをプレビューし、ファイル全体を読むのではなく、不完全な情報が生じる可能性があります。

**悪い例：深すぎる**：

```markdown
# SKILL.md

[advanced.md](advanced.md) を参照してください...

# advanced.md

[details.md](details.md) を参照してください...

# details.md

実際の情報はここにあります...
```

**良い例：1 レベル深い**：

```markdown
# SKILL.md

**基本的な使用**: [SKILL.md の指示]
**高度な機能**: [advanced.md](advanced.md) を参照してください
**API リファレンス**: [reference.md](reference.md) を参照してください
**例**: [examples.md](examples.md) を参照してください
```

すべての参照ファイルは SKILL.md から直接リンクして、必要に応じて Claude が完全なファイルを読むようにしてください。

### 4.5 目次を使用して長い参照ファイルを構造化する

**原則**: 100 行以上の参照ファイルの場合、上部に目次を含める

これにより、部分的な読み取りでプレビューする場合でも、Claude が利用可能な情報の完全な範囲を確認できます。

**例**：

```markdown
# API リファレンス

## 目次

- 認証とセットアップ
- コアメソッド（作成、読み取り、更新、削除）
- 高度な機能（バッチ操作、ウェブフック）
- エラー処理パターン
- コード例

## 認証とセットアップ

...

## コアメソッド

...
```

Claude は必要に応じて完全なファイルを読むか、特定のセクションにジャンプできます。

## 5. 指示の書き方

### 5.1 明確性 (Clarity)

**チェックリスト**:

- [ ] 各ステップが具体的
- [ ] 曖昧な表現を避ける
- [ ] 専門用語に説明を付ける
- [ ] アクションを明確に示す

**良い例**:

```markdown
1. `src/components/` ディレクトリに新しいファイル `Button.tsx` を作成
2. 以下のテンプレートをコピー
3. `[COMPONENT_NAME]` を実際のコンポーネント名に置換
```

**悪い例**:

```markdown
1. 適切な場所にファイルを作成 # どこ？
2. 必要なコードを書く # 何を？
3. うまく調整する # どうやって？
```

### 5.2 構造化

**チェックリスト**:

- [ ] 番号付きリストでステップを示す
- [ ] 関連する情報をグループ化
- [ ] ヘッダーで階層を明確に

**良い例**:

```markdown
## データ処理フロー

### ステップ 1: データ読み込み

1. CSVファイルを読み込む
2. ヘッダー行を検証

### ステップ 2: データ変換

1. 日付フィールドを ISO 8601 形式に変換
2. 欠損値を処理
```

### 5.3 条件分岐の扱い

**チェックリスト**:

- [ ] 条件を明確に記述
- [ ] 各パターンの処理を説明

**良い例**:

```markdown
## 環境別の設定

### 開発環境の場合

`.env.development` を使用:
```

API_URL=http://localhost:3000

```

### 本番環境の場合
`.env.production` を使用:
```

API_URL=https://api.example.com

```

```

### 5.4 エラーハンドリング

**チェックリスト**:

- [ ] 起こりうるエラーを列挙
- [ ] 各エラーの対処法を説明
- [ ] 検証ステップを含める

**例**:

````markdown
## よくあるエラー

### エラー: "Module not found"

**原因**: 依存パッケージがインストールされていない

**解決策**:

```bash
npm install
```
````

### エラー: "Permission denied"

**原因**: ファイルの書き込み権限がない

**解決策**:

```bash
chmod +w target-file.txt
```

## 6. 例の作成

### 6.1 必須要素

**チェックリスト**:

- [ ] 最低 1 つの例を含める
- [ ] 例が実際に動作する
- [ ] 例が具体的で完全

### 6.2 良い例の特徴

**COMPLETE (完全)**:

- [ ] 入力から出力まで全て示す
- [ ] 前提条件を明記
- [ ] 実行可能

**CONCRETE (具体的)**:

- [ ] プレースホルダーではなく実際の値
- [ ] リアルなユースケース

**CLEAR (明確)**:

- [ ] 各ステップを説明
- [ ] なぜそうするのかを示す

**VARIED (多様)**:

- [ ] 異なるユースケース
- [ ] 複雑さのレベルが異なる例

### 6.3 例のテンプレート

```markdown
### 例 [番号]: [ユースケース名]

**目的**: [この例が何を示すか]

**前提条件**:

- [必要な環境や設定]

**シナリオ**:
[具体的な状況説明]

**入力**:
[入力データやコマンド]

**実行手順**:

1. [ステップ 1]
2. [ステップ 2]
3. [ステップ 3]

**出力**:
[期待される結果]

**ポイント**:

- [重要な注意点]
- [学びや応用]
```

### 6.4 コードブロックのベストプラクティス

**チェックリスト**:

- [ ] 言語指定を含める (\`\`\`javascript, \`\`\`python, etc.)
- [ ] インデントを正しく保つ
- [ ] コメントで説明を追加
- [ ] 長すぎる場合は分割

**良い例**:

```typescript
// ユーザーデータの型定義
interface User {
  id: number;
  name: string;
  email: string;
}

// ユーザーを作成する関数
function createUser(name: string, email: string): User {
  return {
    id: Date.now(),
    name,
    email,
  };
}
```

## 7. ワークフローとフィードバックループ

### 7.1 複雑なタスクにはワークフローを使用する

複雑な操作を明確で順序立ったステップに分割してください。特に複雑なワークフローの場合は、Claude が応答にコピーして進行状況をチェックできるチェックリストを提供してください。

**例 1：研究統合ワークフロー**（コードなしのスキル）：

````markdown
## 研究統合ワークフロー

このチェックリストをコピーして進行状況を追跡してください：

```
研究の進行状況：
- [ ] ステップ 1：すべてのソースドキュメントを読む
- [ ] ステップ 2：主要なテーマを特定する
- [ ] ステップ 3：クレームを相互参照する
- [ ] ステップ 4：構造化された要約を作成する
- [ ] ステップ 5：引用を確認する
```

**ステップ 1：すべてのソースドキュメントを読む**

`sources/` ディレクトリ内の各ドキュメントを確認してください。主な議論と支持する証拠に注意してください。

**ステップ 2：主要なテーマを特定する**

ソース全体のパターンを探してください。どのテーマが繰り返し現れますか？ソースが同意または不同意する場所はどこですか？

**ステップ 3：クレームを相互参照する**

各主要なクレームについて、ソース資料に表示されることを確認してください。各ポイントをサポートするソースに注意してください。

**ステップ 4：構造化された要約を作成する**

テーマ別に調査結果を整理してください。以下を含めてください：

- 主なクレーム
- ソースからの支持する証拠
- 矛盾する視点（ある場合）

**ステップ 5：引用を確認する**

すべてのクレームが正しいソースドキュメントを参照していることを確認してください。引用が不完全な場合は、ステップ 3 に戻ってください。
````

この例は、コードを必要としない分析タスクにワークフローがどのように適用されるかを示しています。チェックリストパターンは、複雑で複数ステップのプロセスに機能します。

**例 2：PDF フォーム入力ワークフロー**（コード付きスキル）：

````markdown
## PDF フォーム入力ワークフロー

このチェックリストをコピーして、完了時にアイテムをチェックしてください：

```
タスクの進行状況：
- [ ] ステップ 1：フォームを分析する（analyze_form.py を実行）
- [ ] ステップ 2：フィールドマッピングを作成する（fields.json を編集）
- [ ] ステップ 3：マッピングを検証する（validate_fields.py を実行）
- [ ] ステップ 4：フォームに入力する（fill_form.py を実行）
- [ ] ステップ 5：出力を確認する（verify_output.py を実行）
```

**ステップ 1：フォームを分析する**

実行：`python scripts/analyze_form.py input.pdf`

これはフォームフィールドとその場所を抽出し、`fields.json` に保存します。

**ステップ 2：フィールドマッピングを作成する**

`fields.json` を編集して、各フィールドの値を追加してください。

**ステップ 3：マッピングを検証する**

続行する前に検証エラーを修正してください。

**ステップ 4：フォームに入力する**

実行：`python scripts/fill_form.py input.pdf fields.json output.pdf`

**ステップ 5：出力を確認する**

実行：`python scripts/verify_output.py output.pdf`

検証が失敗した場合は、ステップ 2 に戻ってください。
````

明確なステップにより、Claude が重要な検証をスキップするのを防ぎます。チェックリストは、複数ステップのワークフローを通じて Claude とあなたの両方が進行状況を追跡するのに役立ちます。

## 10. 評価と反復

### 10.1 最初に評価を構築する

**広範なドキュメントを作成する前に評価を作成してください。** これにより、スキルが想像上のものではなく実際の問題を解決することが保証されます。

**評価駆動開発**：

1. **ギャップを特定する**：スキルなしで代表的なタスクで Claude を実行してください。特定の失敗または不足しているコンテキストを文書化してください
2. **評価を作成する**：これらのギャップをテストする 3 つのシナリオを構築してください
3. **ベースラインを確立する**：スキルなしで Claude のパフォーマンスを測定してください
4. **最小限の指示を書く**：ギャップに対処して評価に合格するのに十分なコンテンツのみを作成してください
5. **反復する**：評価を実行し、ベースラインと比較し、改善してください

このアプローチにより、実現しない可能性のある要件を予測するのではなく、実際の問題を解決していることが保証されます。

**評価構造**：

```json
{
  "skills": ["pdf-processing"],
  "query": "この PDF ファイルからすべてのテキストを抽出し、output.txt に保存してください",
  "files": ["test-files/document.pdf"],
  "expected_behavior": [
    "適切な PDF 処理ライブラリまたはコマンドラインツールを使用して PDF ファイルを正常に読み取る",
    "ドキュメント内のすべてのページからテキストコンテンツを抽出し、ページを逃さない",
    "抽出されたテキストを明確で読みやすい形式で output.txt という名前のファイルに保存する"
  ]
}
```

**注意**: この例は、シンプルなテストルーブリックを使用したデータ駆動評価を示しています。現在、これらの評価を実行するための組み込み方法は提供していません。ユーザーは独自の評価システムを作成できます。評価はスキル有効性を測定するための真実の源です。

### 10.2 Claude を使用してスキルを反復的に開発する

最も効果的なスキル開発プロセスは Claude そのものを含みます。1 つの Claude インスタンス（「Claude A」）と協力してスキルを作成し、他のインスタンス（「Claude B」）で使用されます。

**新しいスキルを作成する**：

1. **スキルなしでタスクを完了する**：Claude A で通常のプロンプティングを使用して問題を解決してください。繰り返し提供する情報に注意してください。

2. **再利用可能なパターンを特定する**：タスクを完了した後、同様の将来のタスクに役立つコンテキストを特定してください。

3. **Claude A にスキルを作成するよう依頼する**：「この BigQuery 分析パターンをキャプチャするスキルを作成してください。テーブルスキーマ、命名規則、およびテストアカウントをフィルタリングするルールを含めてください。」

   **ヒント**: Claude モデルはスキル形式と構造をネイティブに理解しています。スキルを作成するように Claude に依頼するために特別なシステムプロンプトや「スキル作成」スキルは必要ありません。

4. **簡潔さについて確認する**：Claude A が不要な説明を追加していないことを確認してください。

5. **情報アーキテクチャを改善する**：Claude A にコンテンツをより効果的に整理するよう依頼してください。

6. **同様のタスクでテストする**：Claude B（スキルがロードされた新しいインスタンス）で関連する使用例でスキルを使用してください。

7. **観察に基づいて反復する**：Claude B が苦労したり、何かを逃したりした場合、具体的に Claude A に戻ってください。

**既存のスキルを反復する**：

同じ階層パターンは、スキルを改善するときに続きます。以下の間で交互に行います：

- **Claude A と協力する**（スキルの改善を支援するエキスパート）
- **Claude B でテストする**（スキルを使用して実際の作業を実行するエージェント）
- **Claude B の動作を観察する**、Claude A に洞察を戻す

**このアプローチが機能する理由**：Claude A はエージェントのニーズを理解し、あなたはドメイン専門知識を提供し、Claude B は実際の使用を通じてギャップを明らかにし、反復的な改善は仮定ではなく観察された動作に基づいてスキルを改善します。

### 10.3 Claude がスキルをナビゲートする方法を観察する

スキルを反復するにつれて、Claude が実際にそれらを使用する方法に注意を払ってください。以下を見てください：

- **予期しない探索パス**：Claude は予期しない順序でファイルを読みますか？
- **接続の欠落**：Claude は重要なファイルへの参照に従うのに失敗しますか？
- **特定のセクションへの過度な依存**：Claude が繰り返し同じファイルを読む場合、そのコンテンツはメインの SKILL.md の代わりにある必要があるかもしれません
- **無視されたコンテンツ**：Claude がバンドルされたファイルにアクセスしない場合、それは不要であるか、メイン指示で不十分に通知されている可能性があります

仮定ではなく、これらの観察に基づいて反復してください。

## 12. 高度：実行可能なコードを含むスキル

以下のセクションは、実行可能なスクリプトを含むスキルに焦点を当てています。スキルがマークダウン指示のみを使用する場合は、「効果的なスキルのチェックリスト」にスキップしてください。

### 12.1 解決する、逃げない

スキルのスクリプトを作成するときは、Claude に逃げるのではなくエラー条件を処理してください。

**良い例：エラーを明示的に処理する**：

```python
def process_file(path):
    """ファイルを処理し、存在しない場合は作成します。"""
    try:
        with open(path) as f:
            return f.read()
    except FileNotFoundError:
        # 失敗する代わりにデフォルトコンテンツでファイルを作成
        print(f"ファイル {path} が見つかりません、デフォルトを作成しています")
        with open(path, 'w') as f:
            f.write('')
        return ''
    except PermissionError:
        # 失敗する代わりに代替案を提供
        print(f"{path} にアクセスできません、デフォルトを使用しています")
        return ''
```

**悪い例：Claude に逃げる**：

```python
def process_file(path):
    # 失敗して、Claude に解決させるだけ
    return open(path).read()
```

構成パラメータも正当化され、文書化される必要があります。

**良い例：自己文書化**：

```python
# HTTP リクエストは通常 30 秒以内に完了します
# より長いタイムアウトは遅い接続を考慮します
REQUEST_TIMEOUT = 30

# 3 回の再試行は信頼性と速度のバランスを取ります
# ほとんどの断続的な失敗は 2 回目の再試行で解決します
MAX_RETRIES = 3
```

**悪い例：マジックナンバー**：

```python
TIMEOUT = 47  # なぜ 47？
RETRIES = 5   # なぜ 5？
```

### 12.2 検証可能な中間出力を作成する

Claude が複雑でオープンエンドのタスクを実行する場合、間違いを犯す可能性があります。「計画-検証-実行」パターンは、Claude が最初に構造化形式で計画を作成し、その計画を実行する前にスクリプトで検証することで、早期にエラーをキャッチします。

**例**：スプレッドシートに基づいて PDF の 50 個のフォームフィールドを更新するように Claude に依頼することを想像してください。検証がなければ、Claude は存在しないフィールドを参照したり、矛盾する値を作成したり、必須フィールドを逃したり、更新を不正に適用したりする可能性があります。

**解決策**：ワークフローパターンを使用しますが、変更を適用する前に検証される中間 `changes.json` ファイルを追加してください。

ワークフローは以下のようになります：分析 → **計画ファイルを作成** → **計画を検証** → 実行 → 確認。

**このパターンが機能する理由**：

- **早期にエラーをキャッチ**：検証は変更を適用する前に問題を見つけます
- **機械検証可能**：スクリプトは客観的な検証を提供します
- **可逆的な計画**：Claude はオリジナルに触れずに計画を反復できます
- **明確なデバッグ**：エラーメッセージは特定の問題を指します

**使用する場合**：バッチ操作、破壊的な変更、複雑な検証ルール、高リスク操作。

**実装のヒント**：「フィールド 'signature_date' が見つかりません。利用可能なフィールド: customer_name、order_total、signature_date_signed」のような特定のエラーメッセージで詳細な検証スクリプトを作成して、Claude が問題を修正するのに役立ちます。

### 12.3 ユーティリティスクリプトを提供する

Claude がスクリプトを書くことができたとしても、事前に作成されたスクリプトには利点があります。

**ユーティリティスクリプトの利点**:

- 生成されたコードより信頼性が高い
- トークンを節約する（コンテキストにコードを含める必要がない）
- 時間を節約する（コード生成は不要）
- 使用全体で一貫性を確保する

**重要な区別**：指示で明確にしてください。Claude は以下のどちらを行うべきか：

- **スクリプトを実行する**（最も一般的）：「フィールドを抽出するには `analyze_form.py` を実行してください」
- **参照として読む**（複雑なロジックの場合）：「フィールド抽出アルゴリズムについては `analyze_form.py` を参照してください」

ほとんどのユーティリティスクリプトでは、実行がより信頼性が高く効率的であるため、実行が推奨されます。

**例**:

````markdown
## ユーティリティスクリプト

**analyze_form.py**: PDF からすべてのフォームフィールドを抽出

```bash
python scripts/analyze_form.py input.pdf > fields.json
```

出力形式:

```json
{
  "field_name": { "type": "text", "x": 100, "y": 200 },
  "signature": { "type": "sig", "x": 150, "y": 500 }
}
```

**validate_boxes.py**: 重複する境界ボックスを確認

```bash
python scripts/validate_boxes.py fields.json
# 戻り値: 「OK」または競合をリスト
```

**fill_form.py**: フィールド値を PDF に適用

```bash
python scripts/fill_form.py input.pdf fields.json output.pdf
```
````

### 12.4 ビジュアル分析を使用する

入力を画像としてレンダリングできる場合、Claude に分析させてください。

````markdown
## フォームレイアウト分析

1. PDF を画像に変換してください:

   ```bash
   python scripts/pdf_to_images.py form.pdf
   ```

2. 各ページ画像を分析してフォームフィールドを特定してください
3. Claude はフィールドの場所とタイプを視覚的に確認できます
````

Claude のビジョン機能はレイアウトと構造を理解するのに役立ちます。

### 12.5 パッケージ依存関係

スキルはコード実行環境で実行され、プラットフォーム固有の制限があります：

- **claude.ai**: npm および PyPI からパッケージをインストールでき、GitHub リポジトリからプルできます
- **Anthropic API**: ネットワークアクセスがなく、ランタイムパッケージのインストールがありません

SKILL.md で必要なパッケージをリストし、コード実行ツールドキュメントで利用可能であることを確認してください。

### 12.6 ランタイム環境

スキルはファイルシステムアクセス、bash コマンド、およびコード実行機能を備えたコード実行環境で実行されます。

**Claude がスキルにアクセスする方法**:

1. **メタデータ事前ロード**: 起動時に、すべてのスキルの YAML フロントマターの名前と説明がシステムプロンプトにロードされます
2. **オンデマンドでファイルを読む**: Claude は必要に応じて bash リードツールを使用してファイルシステムから SKILL.md および他のファイルにアクセスします
3. **スクリプトを効率的に実行**: ユーティリティスクリプトは完全なコンテンツをコンテキストにロードせずに bash で実行できます。スクリプトの出力のみがトークンを消費します
4. **大きなファイルのコンテキストペナルティなし**: 参照ファイル、データ、またはドキュメントは、実際に読まれるまでコンテキストトークンを消費しません

**ベストプラクティス**:

- **ファイルパスが重要**: Claude はファイルシステムのようなスキルディレクトリをナビゲートします。フォワードスラッシュ（`reference/guide.md`）を使用し、バックスラッシュは使用しないでください
- **ファイルに説明的な名前を付ける**: コンテンツを示す名前を使用してください：`form_validation_rules.md`、`doc2.md` ではなく
- **発見のために整理する**: ドメインまたは機能別にディレクトリを構造化してください
  - 良い: `reference/finance.md`、`reference/sales.md`
  - 悪い: `docs/file1.md`、`docs/file2.md`
- **包括的なリソースをバンドルする**: 完全な API ドキュメント、広範な例、大規模なデータセットを含めてください。アクセスされるまでコンテキストペナルティはありません
- **決定論的操作にはスクリプトを優先する**: Claude に検証コードを生成させるのではなく、`validate_form.py` を作成してください
- **実行意図を明確にする**:
  - 「フィールドを抽出するには `analyze_form.py` を実行してください」（実行）
  - 「抽出アルゴリズムについては `analyze_form.py` を参照してください」（参照として読む）
- **ファイルアクセスパターンをテストする**: 実際のリクエストでテストして、Claude がディレクトリ構造をナビゲートできることを確認してください

**例**:

```
bigquery-skill/
├── SKILL.md (概要、参照ファイルを指す)
└── reference/
    ├── finance.md (収益指標)
    ├── sales.md (パイプラインデータ)
    └── product.md (使用分析)
```

ユーザーが収益について質問すると、Claude はスキル.md を読み、`reference/finance.md` への参照を見て、bash を呼び出してそのファイルのみを読みます。sales.md および product.md ファイルはファイルシステムに残り、必要になるまでゼロコンテキストトークンを消費します。このファイルシステムベースのモデルがプログレッシブディスクロージャーを可能にするものです。Claude は各タスクが必要とするものを正確にナビゲートして選択的にロードできます。

### 12.7 MCP ツール参照

スキルが MCP（Model Context Protocol）ツールを使用する場合、「ツールが見つかりません」エラーを避けるために常に完全修飾ツール名を使用してください。

**形式**: `ServerName:tool_name`

**例**:

```markdown
BigQuery:bigquery_schema ツールを使用してテーブルスキーマを取得してください。
GitHub:create_issue ツールを使用して問題を作成してください。
```

ここで:

- `BigQuery` と `GitHub` は MCP サーバー名です
- `bigquery_schema` と `create_issue` はそれらのサーバー内のツール名です

サーバープレフィックスがなければ、特に複数の MCP サーバーが利用可能な場合、Claude はツールの場所を特定するのに失敗する可能性があります。

### 12.8 ツールがインストールされていると仮定しない

パッケージが利用可能であると仮定しないでください。

**悪い例：インストールを仮定**:

```markdown
「pdf ライブラリを使用してファイルを処理してください。」
```

**良い例：依存関係について明示的**:

````markdown
「必要なパッケージをインストールしてください: `pip install pypdf`

その後、それを使用してください:

```python
from pypdf import PdfReader
reader = PdfReader("file.pdf")
```
````

**例 (Python)**:

```python
import os
import re

def validate_filename(filename):
    """ファイル名を検証し、パストラバーサルを防ぐ"""
    # 許可する文字のみ
    if not re.match(r'^[a-zA-Z0-9_.-]+$', filename):
        raise ValueError("Invalid filename")

    # 親ディレクトリへのアクセスを防ぐ
    if '..' in filename:
        raise ValueError("Path traversal detected")

    return filename

# 環境変数から API キーを読み込む
api_key = os.getenv('API_KEY')
if not api_key:
    raise RuntimeError("API_KEY environment variable is required")
```

### 14.3 allowed-tools の慎重な使用

**リスク評価**:

**低リスク**:

- `Read`: 読み取り専用

**中リスク**:

- `Write`: 新規ファイル作成
- `Glob`: ファイル検索

**高リスク**:

- `Edit`: 既存ファイル変更
- `Bash`: 任意のコマンド実行

**推奨**:

```yaml
# 必要最小限のツールのみ
allowed-tools:
  - Read
  - Glob
```
